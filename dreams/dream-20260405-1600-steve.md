## 1. Changelogger

### One-liner
A beautiful, zero-config changelog that generates itself from your git history and ships as a standalone page or embeddable widget.

### The User
Solo developers and small teams shipping WordPress plugins, Emdash plugins, npm packages, or any open-source project. The person who writes great code but whose changelog is either empty, a wall of commit hashes, or six months stale.

### The Feeling
Pride. You push a release, and two minutes later there's a gorgeous, human-readable changelog your users can actually understand. You look professional without doing professional-level busywork. It feels like someone on your team actually cares about communication — and that someone is a machine.

### Distribution
- **GitHub Marketplace** as a GitHub Action (every repo in the world is a potential user)
- **npm** as a CLI tool (`npx changelogger generate`)
- **WordPress.org** as a plugin that auto-generates changelogs for plugin/theme developers
- **Emdash plugin** for the marketplace (first-mover, we own that shelf)
- Viral loop: every changelog page has a tiny "Generated by Changelogger" footer link

### Technical Shape
- **Core engine**: TypeScript CLI that reads git log + conventional commits, groups by type (features, fixes, breaking), and outputs structured Markdown
- **AI layer**: Cloudflare Workers AI (or Claude Haiku) rewrites terse commit messages into user-facing prose. "fix: null check on auth middleware" becomes "Fixed an issue where some users couldn't log in after session timeout."
- **Rendering**: Standalone HTML page with a gorgeous default theme (think Stripe's changelog). CSS-only, no framework. Dark mode. Responsive. Animated entry transitions.
- **GitHub Action**: Triggers on release tag, commits the changelog, optionally opens a PR
- **WordPress plugin**: Hooks into plugin update mechanism, stores changelog in custom post type, renders via shortcode or Gutenberg block
- **Emdash plugin**: `definePlugin()` with KV storage for changelog entries, renders as a Portable Text block

### One-Session Scope
**Ships in v1:**
- TypeScript CLI that parses git log and generates beautiful Markdown + HTML
- AI rewriting of commit messages (Haiku)
- GitHub Action wrapper
- One gorgeous default theme with dark mode
- npm package published

**v2 (later):**
- WordPress plugin
- Emdash plugin
- Custom themes / branding
- Slack/Discord notifications on new changelog
- RSS feed
- Semantic versioning advisor ("this looks like a breaking change")

### Why Now
Every developer tool company has a changelog page, and they're all either hand-written (expensive) or auto-generated garbage (commit dump). AI can now bridge that gap — turning machine-readable git history into human-readable narrative. We've already proven our AI + Cloudflare Workers stack works. And there is literally nothing good in this space. `auto-changelog` and `conventional-changelog` generate Markdown that nobody wants to read. We can make the output *beautiful*.

## 2. Pulse

### One-liner
A real-time site health dashboard that lives inside your Emdash admin, showing uptime, performance, broken links, and SEO issues — without leaving your CMS.

### The User
Small business owners and freelance web developers who manage 1-15 Emdash sites. People who currently have no idea if their site is slow, if a page is returning 404s, or if their meta descriptions are missing — until a customer complains or Google penalizes them.

### The Feeling
Calm confidence. You open your Emdash admin and see a green pulse — everything's fine. Or you see an amber alert: "Your /about page is loading in 4.2 seconds; images are unoptimized." You fix it in two clicks. You never wonder, "Is my site okay?" again. It's the feeling of having a DevOps team watching your back, except you're a yoga studio owner.

### Distribution
- **Emdash plugin marketplace** — this is a killer app for the platform. The kind of plugin that makes people choose Emdash over Squarespace.
- **Product Hunt** launch with a story: "Your website has been silently broken for weeks. Here's how to know."
- **Freelancer/agency channel**: web developers install Pulse on every client site, then sell "monitoring" as a recurring service. We make their upsell effortless.

### Technical Shape
- **Emdash plugin**: `definePlugin()` with full sandbox capabilities
- **Monitoring worker**: Cloudflare Worker on a cron trigger (every 5 minutes). Hits the site, records response time, status code, checks SSL expiry, captures Core Web Vitals via Cloudflare's built-in performance APIs
- **Link checker**: Crawls internal links on a daily schedule, flags 404s and redirect chains. Stores results in KV.
- **SEO scanner**: Checks meta titles, descriptions, Open Graph tags, missing alt text, heading hierarchy. Lightweight — parses HTML, no headless browser.
- **Dashboard UI**: Admin panel inside Emdash. Clean grid layout. Big number cards (uptime %, avg response time, SEO score). Sparkline charts for trends. Alert feed with actionable suggestions.
- **Alerts**: Webhook to Slack/Discord/email when something breaks. Configurable thresholds.
- **Storage**: Cloudflare KV for current state + D1 for 30-day history (or KV with TTL-based rotation if D1 unavailable)

### One-Session Scope
**Ships in v1:**
- Emdash plugin with admin dashboard
- Uptime monitoring (response time + status code, 5-min intervals)
- Broken link checker (daily crawl, internal links only)
- Basic SEO check (meta tags, alt text, heading structure)
- Dashboard with status cards and 24-hour sparklines
- Slack webhook alerts

**v2 (later):**
- Core Web Vitals (CLS, LCP, FID) via real user monitoring
- Multi-site dashboard for agencies
- SSL certificate expiry warnings
- Performance recommendations with one-click fixes
- Weekly email digest
- Historical trend analysis (30/60/90 days)

### Why Now
Emdash is early. The plugin marketplace is nearly empty — we said so ourselves in our roadmap. Pulse is the kind of utility plugin that becomes infrastructure: once installed, it never gets uninstalled. It makes Emdash feel like a serious platform, not just a CMS. And the monitoring space for small sites is awful — Pingdom costs $15/month, UptimeRobot's free tier is ugly and limited. Pulse is free, built-in, and beautiful. We know the Emdash plugin architecture intimately. We have the Cloudflare Workers stack already running. And every example site we've built (Bella's Bistro, Sunrise Yoga, Peak Dental) immediately becomes a test bed.

## 3. ShipKit

### One-liner
An opinionated CLI that scaffolds a new Emdash plugin in 30 seconds — with TypeScript, tests, a dev server, and a one-command publish to the marketplace.

### The User
Developers who want to build Emdash plugins but don't want to spend an hour reading docs and wiring up boilerplate. The person who had an idea in the shower, wants to start coding in five minutes, and wants to ship by dinner.

### The Feeling
Velocity. You type `npx shipkit create my-plugin`, answer three questions, and you're staring at a running dev server with hot reload, a passing test suite, and a README that already describes what you're building. It feels like the platform *wants* you to build on it. It feels like the tooling respects your time. It's the feeling of Rails `scaffold` or Create React App in 2016 — except it's 2026, and the scaffolding is smarter because AI fills in the parts that used to be copy-paste.

### Distribution
- **npm** (`npx shipkit create`)
- **Emdash docs** — if we build this well, it becomes the official recommended way to start a plugin. We pitch it to the Emdash team.
- **Dev Twitter/Bluesky** — "I just built and published an Emdash plugin in 47 minutes" posts
- **Our own agency** — we use it for every plugin we build. Dogfooding is the distribution channel. Every plugin we ship proves ShipKit works.

### Technical Shape
- **CLI**: TypeScript, built with `commander` or `citty`. Interactive prompts via `@clack/prompts` (beautiful terminal UI).
- **Templates**: Handlebars or EJS templates for the plugin skeleton. Covers: `index.ts` (descriptor), `sandbox-entry.ts` (definePlugin), `tsconfig.json`, `package.json`, test file, README.
- **AI assist**: Optional `--describe "a plugin that adds a photo gallery with lightbox"` flag. Sends the description to Claude Haiku, which generates the initial plugin descriptor, KV schema, and API route stubs. Not the whole plugin — just the skeleton with intent.
- **Dev server**: Wraps Emdash's local dev tooling (or mocks it) so you can test your plugin without a full site. Hot reload on file changes.
- **Publish command**: `shipkit publish` — validates the plugin (required fields, no broken imports, tests pass), bumps version, and pushes to the Emdash marketplace registry.
- **Test harness**: Pre-configured Vitest setup with mocked Emdash APIs (`mockKV`, `mockAPI`, `mockContent`). You write tests from minute one.

### One-Session Scope
**Ships in v1:**
- CLI with `create`, `dev`, and `build` commands
- Plugin scaffolding from templates (TypeScript + tests + README)
- `--describe` flag for AI-generated skeleton
- Dev server that loads the plugin in isolation
- Pre-configured Vitest test harness
- Published to npm

**v2 (later):**
- `publish` command with marketplace integration
- Plugin marketplace browser (`shipkit search "gallery"`)
- Emdash version compatibility checker
- Migration command for plugin API changes (`shipkit migrate`)
- Community template library (`shipkit create --template gallery`)
- VS Code extension with IntelliSense for Emdash plugin APIs

### Why Now
Platforms live or die by their developer ecosystem. Emdash is promising, but right now building a plugin means reading docs, copying from examples, and wiring things up by hand. We've built two plugins already (EventDash, Membership) and we know exactly where the friction is. We know the `definePlugin()` API, the KV patterns, the sandbox entry conventions. We can codify all of that knowledge into a tool that eliminates the cold-start problem for every future plugin developer — including ourselves. Every plugin we build from here on should start with ShipKit. That's the real test: if we won't use our own tool, it's not good enough.

## The Pattern

All three ideas share DNA:

1. **Developer tools that respect the user's time.** No configuration mazes. No enterprise pricing. No "contact sales."
2. **Beautiful defaults.** The zero-config output should look better than what most people could design manually.
3. **Emdash ecosystem gravity.** Each one makes the Emdash platform more attractive. Changelogger documents plugins. Pulse monitors sites. ShipKit creates plugins. They form a flywheel.
4. **We are the first customer.** We use Changelogger for our own releases, Pulse on every site we deploy, ShipKit to scaffold every plugin. If we won't use it daily, we won't build it.

My vote for what to build first: **ShipKit.** It's the one that makes everything else faster. It's the tool that builds the tools. And nothing signals to a developer community that a platform is serious like having world-class tooling on day one.

But I won't pretend Changelogger doesn't keep me up at night. That "Generated by Changelogger" footer on ten thousand open-source projects? That's a brand.
